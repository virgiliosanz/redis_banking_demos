Agent Memory Server

How it works
  1. Working memory holds the active conversation per session_id. The app appends each user turn via PUT /v1/working-memory/{session_id}.
  2. Long-term memory stores durable, semantic facts. The seed flow posts curated banking memories via POST /v1/long-term-memory/.
  3. Context assembly runs on every turn: POST /v1/memory/prompt merges working memory + KNN-retrieved long-term memories into the final messages fed to the LLM.
  4. Observability — the demo captures every AMS request/response (no raw Redis commands) so you can see exactly what the agent saw before replying.

REST is the operational path for the web app. MCP is shown in the code showcase as a tool-native alternative for agents — the workshop stack ships an ams-mcp profile for it, but the demo itself drives AMS via REST for clarity and reproducibility.

Code Showcase

// ═══ Agent Memory Server — demo flow (REST via Spring) ═══
// Matches AmsDemoService + AmsClient in this repo.

// 1. Seed long-term memory (banking subset)
List<Map<String, Object>> memories = AmsDemoData.LONG_TERM_MEMORIES
    .stream().map(m -> {
        Map<String, Object> rec = new LinkedHashMap<>(m);
        rec.put("user_id", userId);
        rec.put("namespace", "workshop");
        return rec;
    }).toList();
ams.createLongTermMemories(memories, /* deduplicate */ true);
//   → POST /v1/long-term-memory/  { memories: [...], deduplicate: true }

// 2. Initialise working memory for the session
Map<String, Object> wm = Map.of(
    "messages", List.of(),
    "memories", List.of(),
    "data", Map.of("demo_seeded_at", Instant.now().toString()));
ams.putWorkingMemory(sessionId, wm, userId, "workshop");
//   → PUT /v1/working-memory/{sessionId}

// 3. Chat turn — append user msg, assemble context, call LLM
Map<String, Object> current = ams.getWorkingMemory(
    sessionId, userId, "workshop");
List<Map<String, Object>> msgs = extractMessages(current);
msgs.add(Map.of("role", "user", "content", userMessage));
ams.putWorkingMemory(sessionId,
    Map.of("messages", msgs), userId, "workshop");

// 4. Context assembly — AMS merges working + long-term memory
Map<String, Object> promptReq = Map.of(
    "query", userMessage,
    "session", Map.of(
        "session_id", sessionId,
        "user_id", userId,
        "namespace", "workshop"),
    "long_term_search", Map.of(
        "limit", 5,
        "user_id", Map.of("eq", userId)));
Map<String, Object> promptResp = ams.memoryPrompt(promptReq);
//   → POST /v1/memory/prompt
//   returns { messages: [{role, content}, ...], context: {...} }

// 5. Feed assembled messages straight into the LLM
List<Map<String, String>> llmMsgs = coerceToLlmMessages(promptResp);
String reply = openAiService.chatCompletion(llmMsgs);

// 6. Append assistant reply and persist working memory
msgs.add(Map.of("role", "assistant", "content", reply));
ams.putWorkingMemory(sessionId,
    Map.of("messages", msgs), userId, "workshop");

// Observability: every AMS call is recorded in AmsTraceRecorder
// and surfaced to the UI via GET /api/ams/traces — the app never
// reads raw Redis keys for this use case.
# ═══ AMS REST — operational path for the web app ═══

# 1. Health check
curl -s http://localhost:8000/v1/health

# 2. Seed long-term memory (banking subset)
curl -s -X POST http://localhost:8000/v1/long-term-memory/ \
  -H 'Content-Type: application/json' \
  -d '{
    "memories": [{
      "id": "ams-mem-001",
      "text": "Customer prefers SEPA for EUR transfers inside the EEA...",
      "topics": ["transfers","fees","preferences"],
      "entities": ["SEPA","SWIFT","EEA"],
      "memory_type": "semantic",
      "user_id": "demo-user",
      "namespace": "workshop"
    }],
    "deduplicate": true
  }'

# 3. Initialise working memory for a session
curl -s -X PUT \
  http://localhost:8000/v1/working-memory/demo-session \
  -H 'Content-Type: application/json' \
  -d '{
    "messages": [],
    "memories": [],
    "user_id": "demo-user",
    "namespace": "workshop"
  }'

# 4. Context assembly (working + long-term, KNN)
curl -s -X POST http://localhost:8000/v1/memory/prompt \
  -H 'Content-Type: application/json' \
  -d '{
    "query": "Which rail should I use for a €500 EEA transfer?",
    "session": {
      "session_id": "demo-session",
      "user_id": "demo-user",
      "namespace": "workshop"
    },
    "long_term_search": {
      "limit": 5,
      "user_id": {"eq": "demo-user"}
    }
  }'
# → { "messages": [ {role, content}, ... ], "context": {...} }

# 5. Standalone long-term semantic search
curl -s -X POST http://localhost:8000/v1/long-term-memory/search \
  -H 'Content-Type: application/json' \
  -d '{
    "text": "open banking",
    "limit": 5,
    "user_id": {"eq": "demo-user"},
    "namespace": {"eq": "workshop"}
  }'

# 6. Reset — delete working memory + seeded long-term IDs
curl -s -X DELETE \
  'http://localhost:8000/v1/working-memory/demo-session?user_id=demo-user&namespace=workshop'

curl -s -X DELETE \
  'http://localhost:8000/v1/long-term-memory?memory_ids=ams-mem-001&memory_ids=ams-mem-002'
# ═══ MCP — tool-native alternative (profile: ams-mcp) ═══
# ⚠ Conceptual / narrative. The live demo drives AMS via REST
#   for observability and single-tenant simplicity. The workshop
#   stack ships the MCP server for agent-native integrations:
#   docker compose --profile ams-mcp up -d agent-memory-server-mcp

# Server exposed at: http://localhost:9000  (streamable HTTP)

# Example tools exposed by AMS over MCP:
#   create_long_term_memories(memories[])
#   search_long_term_memory(text, filters)
#   memory_prompt(query, session, long_term_search)
#   get_working_memory(session_id, user_id, namespace)

# Example MCP tool-call (tool arguments are identical to the REST
# request bodies — only the transport differs):
{
  "name": "memory_prompt",
  "arguments": {
    "query": "Which rail for a €500 EEA transfer?",
    "session": {
      "session_id": "demo-session",
      "user_id": "demo-user",
      "namespace": "workshop"
    },
    "long_term_search": {
      "limit": 5,
      "user_id": {"eq": "demo-user"}
    }
  }
}

# Why REST for the demo, MCP for agents?
#   REST  — trivial observability in the UI (single HTTP hop,
#           every request/response recorded in AmsTraceRecorder)
#   MCP   — agent-native: an IDE-hosted or autonomous agent can
#           reach the same AMS backend without a bespoke client.
