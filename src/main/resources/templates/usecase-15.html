UC15

AI Guardrails for Banking Chat

How it works
  1. Rate limit uses Redis INCR + EXPIRE per user to enforce a 60-second token budget.
  2. Topic routing embeds the prompt with a mock 1536-dim vector and runs FT.SEARCH ... KNN against Redis route vectors.
  3. PII and injection checks flag sensitive values and block jailbreak-style prompts before the mock LLM runs.
  4. Output guardrails scrub leaked identifiers and add an investment disclaimer when the reply needs compliance hardening.
  5. Audit trail writes every decision to a Redis Stream so the UI can replay the pipeline step-by-step.

// UC15 — input guardrails
Long count = redis.opsForValue().increment("uc15:rate:" + userId);
redis.expire("uc15:rate:" + userId, 60, TimeUnit.SECONDS);

byte[] blob = RedisSearchHelper.vectorToBytes(buildMockVector(message));
List<Object> raw = redisSearchHelper.ftSearchWithBinaryArgs(
    "idx:uc15:routes",
    new byte[][] {
        "*=>[KNN 1 @vector $BLOB AS score]".getBytes(),
        "RETURN".getBytes(), "3".getBytes(),
        "label".getBytes(), "action".getBytes(), "score".getBytes(),
        "PARAMS".getBytes(), "2".getBytes(), "BLOB".getBytes(), blob,
        "DIALECT".getBytes(), "2".getBytes()
    }
);

redis.opsForStream().add(StreamRecords.string(Map.of(
    "stage", "topic", "status", "PASS", "latencyMs", "3"
)).withStreamKey("uc15:stream:audit"));
# Rate limiting
INCR uc15:rate:demo-user
EXPIRE uc15:rate:demo-user 60

# Topic routing with Redis Query Engine
FT.SEARCH idx:uc15:routes "*=>[KNN 1 @vector $BLOB AS score]" \
  RETURN 3 label action score \
  SORTBY score PARAMS 2 BLOB <vector-bytes> DIALECT 2

# Prompt injection detection
FT.SEARCH idx:uc15:injections "*=>[KNN 1 @vector $BLOB AS score]" \
  RETURN 3 pattern severity score \
  SORTBY score PARAMS 2 BLOB <vector-bytes> DIALECT 2

# Audit trail
XADD uc15:stream:audit * stage rateLimit status PASS latencyMs 1 detail "Within budget"
XRANGE uc15:stream:audit - + COUNT 20
